Overview

OpenStack Barbican is the key manager service for OpenStack. It provides secure storage, provisioning, and management of secrets such as passwords, API keys, encryption keys, and X.509 certificates.

This UI provides a browser-based interface covering all Barbican operations that the openstack secret CLI offers. Think of it as a web-based equivalent of the CLI, with additional features like a graphical JSON editor for structured secrets.

Key principle: Barbican stores secrets server-side. This UI never persists secrets locally — everything is fetched from and stored via the Barbican API. The only local storage is a disk-based cache of metadata to reduce API calls.
{# ── Secrets ───────────────────────────────────────────────── #}

Secrets

A secret is the fundamental resource in Barbican. It can hold any sensitive data: passwords, API tokens, SSH keys, certificates, encryption keys, or arbitrary binary data.

Secret Lifecycle
  1. Store — create a new secret with a name and payload
  2. Get — retrieve the secret's metadata and/or its payload
  3. Update — set or replace the payload (only if the secret was created without one)
  4. Delete — permanently remove the secret
Important: Once a secret's payload is set, it cannot be changed. The "update" operation in Barbican only works on secrets that were created without a payload (two-step creation). To change a secret's value, you must delete it and create a new one.
Metadata Fields
FieldDescription
nameHuman-readable name. Use / separators for virtual folder organization (e.g. production/db/password).
secret_typeCryptographic classification. See Secret Types.
algorithmAlgorithm used (e.g. aes, rsa). Informational only.
bit_lengthKey size in bits (e.g. 256). Informational only.
modeEncryption mode (e.g. cbc). Informational only.
expirationISO 8601 timestamp after which Barbican may auto-delete the secret.
statusACTIVE (has payload) or PENDING (no payload yet).
User Metadata

Each secret supports user metadata — editable key-value tags that can be added, modified, and removed at any time. Unlike core metadata (name, type, algorithm), user metadata is mutable. Use it for labels, environment tags, ownership info, or any custom annotations.

Where to find: Secret detail page → User Metadata card → click Add.

Clone & Replace

Since Barbican secrets are immutable (core metadata and payload cannot be changed after creation), this UI provides two workflows for "editing":

Replace

Deletes the original secret and creates a new one with the same name.

  • Name and path are locked (greyed out)
  • Original secret is deleted
  • New secret gets a new ID
  • Effectively "edits" the secret
Clone

Creates a new secret with a different name (-copy suffix), pre-filled with all data from the original.

  • Name and path are editable
  • Original secret is kept
  • Useful for creating variants or backups
Replace changes the secret ID. Any external systems referencing the old secret ID (e.g. container references, External Secrets Operator configs) will need to be updated. If your references use secret names rather than IDs, this is not an issue.
{# ── Secret Types ──────────────────────────────────────────── #}

Secret Types

Barbican classifies secrets by their intended cryptographic purpose:

TypeUse CaseExamples
opaqueGeneral purpose — any arbitrary dataPasswords, API keys, config files, JSON blobs
symmetricSymmetric encryption keysAES-256 key, HMAC key
publicPublic key of an asymmetric pairRSA public key, EC public key
privatePrivate key of an asymmetric pairRSA private key, EC private key
passphrasePassphrase for key protectionPrivate key passphrase
certificateX.509 certificateTLS/SSL certificates
For most use cases (passwords, API keys, JSON configs), use opaque. The other types are meaningful when Barbican interacts with crypto backends (HSMs) or when you use containers to group related cryptographic material.
{# ── Virtual Folders ───────────────────────────────────────── #}

Virtual Folders

Barbican has no native folder or namespace support. All secrets are stored in a flat list within a project/tenant.

This UI adds virtual folders by using / in secret names as a path separator. For example:

production/database/mysql-password
production/database/redis-url
production/api/stripe-key
staging/database/mysql-password

This creates a browsable tree:

  • production
    • database → mysql-password, redis-url
    • api → stripe-key
  • staging
    • database → mysql-password

When creating a secret, use the Path field to set the folder. The field offers autocomplete with existing paths. Secrets without / in the name appear at the root level.

{# ── Payload Modes ─────────────────────────────────────────── #}

Payload Modes

When creating a secret, you can choose between three payload editing modes:

Simple

Plain text or binary payload. Enter the raw value directly.

Best for: single passwords, tokens, certificates (PEM format).

Key-Value

Add multiple key-value pairs using a dynamic form. Stored as JSON.

Best for: database credentials, connection strings, config groups.

JSON

Full graphical JSON editor with tree, code, and form views. Supports nested structures.

Best for: complex configs, nested data, structured secrets (similar to HashiCorp Vault).

Key-Value and JSON modes both store the payload as a JSON string with content type text/plain. When viewing the secret, the UI automatically detects JSON payloads and displays them in both table (key-value) and tree (JSON editor) views.
{# ── Containers ────────────────────────────────────────────── #}

Containers

A container is a logical grouping of related secrets by reference. It does not store data itself — it holds links (URLs) to existing secrets.

Containers are not folders. They don't create hierarchy. Think of them as playlists: deleting a container does not delete the secrets it references. The same secret can be in multiple containers.
Container Types
TypePurposeExpected References
generic Arbitrary group of secrets Any names — you define the reference names
rsa RSA key pair public_key, private_key, optionally private_key_passphrase
certificate TLS/SSL certificate bundle certificate, private_key, optionally intermediates, private_key_passphrase
Typical Use Case

You have a TLS certificate setup with 3 secrets: the certificate itself, the private key, and the CA chain. Instead of managing 3 separate secret IDs, you create a certificate container that references all three. Consumers (like a load balancer) can then fetch the entire bundle via one container reference.

{# ── Consumers ─────────────────────────────────────────────── #}

Consumers

A consumer is a registration record that tracks who or what is using a container. Consumers are always attached to a container, not to individual secrets.

How Consumers Work
  1. An application registers itself as a consumer of a container, providing a name and a URL
  2. Barbican records the registration
  3. If someone tries to delete the container, Barbican blocks the deletion if consumers are registered
  4. To delete the container, all consumers must be removed first
Why This Matters
  • Protection: prevents accidental deletion of secrets that are in active use
  • Audit trail: you can see which services depend on which secrets
  • Coordination: before rotating a certificate, check who is consuming it
Where to find consumers in this UI: Open any container's detail page → scroll down to the "Consumers" section. You can register and remove consumers there.
{# ── Orders ────────────────────────────────────────────────── #}

Orders

An order is a request for Barbican to generate cryptographic material server-side, rather than you creating and uploading it. This is more secure because the key material is generated inside Barbican's backend (potentially an HSM) and never travels over the network.

Order Types
TypeWhat It GeneratesResult
key Symmetric key (e.g. AES-256) A single secret with the generated key
asymmetric Asymmetric key pair (e.g. RSA-2048) A container with public_key and private_key secrets
certificate X.509 certificate Depends on CA plugin configuration
Order Lifecycle
PENDING ACTIVE (success) or ERROR (failure)

After creating an order, Barbican processes it asynchronously. Once complete, the order's secret_ref field points to the generated secret (or container for asymmetric keys).

When to Use Orders vs. Direct Store
ScenarioUse
You already have a password/key/certSecret Store — upload it directly
You need a new encryption keyOrder — let Barbican generate it securely
You need an RSA key pairOrder (asymmetric) — generates both keys at once
{# ── Transfer ───────────────────────────────────────────────── #}

Export / Import (Transfer)

The Transfer page lets you migrate secrets and containers between Barbican instances, regions, or projects — by exporting to a JSON file and importing it elsewhere.

Export

Downloads a JSON file containing:

  • Secrets — name, type, algorithm, metadata, and optionally the plaintext payload
  • Containers — name, type, and secret references (resolved by secret name, not ID)
  • Consumers — name and URL for each consumer registered on a container
Security: If you include payloads, the exported file contains plaintext secret values. Treat the file with the same care as the secrets themselves. Do not commit it to version control or share it over insecure channels.
Import

Uploads a previously exported JSON file and recreates the resources:

  1. Secrets are created first
  2. Containers are created next — secret references are resolved by matching the secret name to the newly created secret's ID
  3. Consumers are re-registered on the new containers
Options
OptionDescription
Skip existing If a secret with the same name already exists in the target instance, it is skipped (not overwritten). The existing secret's ID is used for container references. Enabled by default.
Dry run Simulates the import and reports what would be created, without actually creating anything. Use this to preview before committing.
Use Cases
ScenarioHow
Migrate secrets from region A to region B Log in to region A → Export → Log out → Log in to region B → Import
Backup all secrets Export with payloads → store the JSON file securely
Clone secrets to a staging project Export from production project → Log in to staging project → Import with "skip existing"
Audit secret inventory Export without payloads → review the JSON metadata
File Format

The export file is a versioned JSON document ("version": "1.0") containing:

  • exported_at — timestamp of the export
  • source_endpoint — the Barbican API URL it was exported from
  • project_id / project_name — source project info
  • secrets[] — array of secret objects with optional payload field
  • containers[] — array of containers with secret_refs using secret names (not UUIDs) and consumers array
Why names instead of IDs? Secret UUIDs are unique per Barbican instance. When importing into a different instance, new UUIDs are generated. By referencing secrets by name, container relationships are preserved across instances.
{# ── Permissions ───────────────────────────────────────────── #}

Required Permissions

This application uses the minimum possible OpenStack permissions:

ServiceEndpointPermission
Keystone POST /v3/auth/tokens Any valid user (standard authentication)
Barbican /v1/secrets/*, /v1/containers/*, /v1/orders/* Project-scoped creator role
No admin access required. No access to Nova, Neutron, Glance, or any other OpenStack service. The creator role on the key-manager service is sufficient for all operations.

To grant access:

openstack role add --project <project-id> --user <user-id> creator
